# OpenClaw Project Highlights & User Guide

Based on the architecture of OpenClaw (formerly Moltbot) and the provided reference material, here are the key highlights and answers to your specific questions about Memory and Skills.

## 1. Project Highlights (The "Twitter Points")

These are the technical differentiators that make OpenClaw unique compared to standard "wrapper" AI apps:

### 🏗️ Architecture: Process over App
- **TypeScript CLI:** It's not a Python script or a Next.js web app. It runs as a persistent **process** on your machine.
- **Gateway Server:** The core is a task coordinator that manages connections to all channels (Telegram, Slack, etc.).
- **Lane-Based Serial Queues:** Instead of "async/await spaghetti," OpenClaw uses **serial lanes** by default. This prevents race conditions when multiple agents or tools try to act simultaneously. You don't ask "what do I lock?", you ask "what is safe to parallelize?".

### 🧠 Hybrid Memory System
- **Two-Tier Storage:**
  1. **JSONL:** Raw, complete session logs.
  2. **Markdown:** Synthesized, readable memory files (`MEMORY.md` or `memory/*.md`).
- **Hybrid Search:** Combines **Vector Search** (semantic) with **Keyword Search** (FTS5). This means it finds things whether you use the exact word or just describe the concept.
- **Smart Syncing:** Watches for file changes to update its index in real-time.

### 🖥️ Computer Use (The "Moat")
- **Semantic Snapshots:** When browsing the web, it doesn't just take screenshots (expensive, slow). It reads the **Accessibility Tree** (ARIA) to create a text-based "semantic snapshot" of the page. This is faster and uses fewer tokens than vision models.
- **Sandboxing:** Capable of running commands in Docker containers for safety, or on the host machine for power.
- **Process Management:** Can run background processes and manage them, unlike simple "request-response" bots.

## 2. Memory: "How should I store memory?"

**Your Question:** *Currently I use a section to store memory, that is, after each conversation ends, let the system store my memory. Is there any other way?*

**Analysis of OpenClaw's Approach:**
You are on the right track, but OpenClaw suggests a more robust lifecycle:

1.  **The "New Session" Hook:**
    Instead of waiting for the *end* of a chat (which might crash or be abandoned), OpenClaw uses a **Hook** that triggers when a *new* session starts (`/new` command).
    *   *How it works:* It reads the *previous* session's transcript, summarizes it into a Markdown file with a descriptive slug (generated by an LLM), and saves it to `memory/`.
    *   *Reference:* See `src/hooks/bundled/session-memory/handler.ts`.

2.  **Pre-Compaction Flush:**
    For very long conversations, you shouldn't wait until the end. OpenClaw has a **Memory Flush** mechanism.
    *   *How it works:* When the context window gets full and needs "compaction" (summarizing to save space), the system first asks the agent: "Store durable memories now... If nothing to store, reply with SILENT_REPLY."
    *   *Benefit:* This saves key details *before* they get compressed and potentially lost.

**Recommendation:**
*   **Don't just wait for the end.** Configure your system to allow the agent to write to memory *during* the conversation if it learns a new fact about you (e.g., "User moved to a new address").
*   **Use a "Flush" trigger.** If a conversation exceeds a certain token limit, trigger a "memory save" step automatically.

## 3. Skills: "How to find better skills?"

**Your Question:** *How can I find better skills?*

**Analysis:**
In OpenClaw, "Skills" are essentially **Tools** bundled with documentation.

1.  **ClawHub (The Registry):**
    The project points to **[ClawHub](https://clawhub.com)** as the official registry for finding new skills. This is the "App Store" for OpenClaw capabilities.

2.  **Structure (`SKILL.md`):**
    Look at the `skills/` directory in the root. Each skill (like `1password`, `github`) is defined by a `SKILL.md` file.
    *   This file tells the agent *how* to use the tools and when.
    *   It often maps to CLI commands or API calls.

3.  **Model Context Protocol (MCP):**
    The industry is moving towards the **Model Context Protocol (MCP)** (by Anthropic). OpenClaw's architecture allows it to adapt these standard tools.
    *   *Advice:* Look for "MCP Servers" or "MCP Tools" on GitHub. Many high-quality "skills" are being built in this format, which can be adapted to OpenClaw.

**Best Advice:**
The "best" skills are often the ones you write yourself for your specific workflow. Because OpenClaw runs as a process, you can write a simple script (Shell, Python, TS) and expose it as a tool in a `SKILL.md` file. This gives the agent control over *your* specific apps and data.
